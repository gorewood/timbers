The hooks story this session was a proper arc — one of those multi-hour journeys where you start fixing a small thing and end up rethinking an entire subsystem.

It started with a discovery: the plain-text echo hooks timbers had been installing for Claude Code? **Silently ignored.** Every session, commits went undocumented because Claude Code just... ate the stdout and moved on. The only mechanism it actually respects for blocking is structured JSON with `permissionDecision/deny`. So the first order of business was rebuilding `PreToolUse` and `Stop` as proper JSON-speaking hooks, with all the logic living in the `timbers` binary itself.

That worked. Then it immediately broke.

`HasPendingCommits()` was designed as a fast ~15ms check — just compare HEAD against the anchor commit. The original plan even had a note: *"may false-positive on ledger-only commits; acceptable trade-off for ~15ms."* Testing in a real repo proved that wrong in about ten seconds. Every `timbers log` auto-commits an entry, which advances HEAD, which makes `HasPendingCommits` say "yes!" — which blocks every single session end. The fast path saved 85ms but made the Stop hook completely useless. Ripped that out and delegated to `GetPendingCommits` with proper ledger-commit filtering. One extra git subprocess per session. Fine.

Then came the realization that `PreToolUse` was redundant. Pre-commit hooks block universally across *all* git clients — not just Claude Code. The JSON protocol negotiation for `PreToolUse` was adding real complexity for marginal gain. Dropped it entirely, kept `Stop` as a backstop, moved `PreToolUse` to `retiredEvents`. A surprisingly hefty chunk of plumbing scattered across the codebase, but the result is cleaner.

> Which surfaced another bug: `removeTimbersHooks` only iterated *current* events. Anyone upgrading from v0.12/v0.13 would have orphaned `PostToolUse`/`PreToolUse` hooks forever. Added a `retiredEvents` loop and a test.

Meanwhile, `core.hooksPath` was quietly sabotaging everything. Beads sets `core.hooksPath=.beads/hooks/` and timbers was hardcoding `.git/hooks/` — so hooks were being installed where git literally never looks. A tiny fix to `GetHooksDir` (read the config, fall back to `.git/hooks`) patched it, but I knew it was a band-aid.

The real fix landed as v0.15.0: **graceful hook deference**. A four-tier environment classification — uncontested, existing hooks, known override (`core.hooksPath`), unknown override — with an append-section strategy instead of the old backup-and-chain approach. The key insight from the design session: rather than maintaining `.original` backups and a second code path for symlinks and binaries, just refuse those cases with guidance and append a section to normal hook files. The debates were illuminating — whether to prompt during `init` for Tier 1 repos (no, steering via Stop hook is primary enforcement, git hooks are a bonus), `.original` vs `.backup` naming (moot once append-section eliminated the need). A new `hooks status` command makes the whole thing inspectable.

Oh, and the batch performance fix: `filterLedgerOnlyCommits` was spawning one git subprocess per commit. In a 2k-commit repo, `doctor` took **16 seconds**. `CommitFilesMulti` using `git diff-tree --stdin` collapsed that to a single process. The kind of fix where the before/after is viscerally satisfying.

Also explored co-committing entries with code — where the entry and the code it documents share the same commit. Went deep on ~12 approaches (two-pass commits, predictable IDs, post-commit amendment, deferred IDs). They all break something. The entry ID contains the anchor commit SHA, so co-committing creates a circular dependency: entry changes the tree, which changes the SHA, which changes the entry. Separate commits is the right design. The SHA reference is a feature.
